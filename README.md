# Smia (The Forge)

Smia is a desktop configuration for AerynOS. It uses Malm to build and deploy
the configuration, while Gnist handles themes at runtime.

The available compositor setups are Mango, Niri, and Hyprland. This repository
is still a work in progress and is mainly intended for the system it was built
on.

## Install

Bootstrap the installer from the nested system-model configuration. Preview the
two generated commands first:

```sh
malm \
    --config system-models/malm.kdl \
    --state smia-system-models \
    --profile install \
    plan https://github.com/christian-bendiksen/smia.git \
    --branch main \
    --allow-local-includes
```

Apply and track that state when the plan looks correct:

```sh
malm \
    --config system-models/malm.kdl \
    --state smia-system-models \
    --profile install \
    apply https://github.com/christian-bendiksen/smia.git \
    --branch main \
    --trust-remote \
    --track \
    --allow-local-includes
```

The bootstrap state owns the `smia` dispatcher and its `install` plugin. Choose
one compositor and either the desktop or gaming kernel for the complete install:

| Model profile | Compositor | Kernel |
|---|---|---|
| `mango-desktop` | Mango | `linux-desktop` |
| `mango-gaming` | Mango | `linux-gaming` |
| `niri-desktop` | Niri | `linux-desktop` |
| `niri-gaming` | Niri | `linux-gaming` |
| `hyprland-desktop` | Hyprland | `linux-desktop` |
| `hyprland-gaming` | Hyprland | `linux-gaming` |

For example:

```sh
smia install niri-gaming --plan
smia install niri-gaming
```

The installer previews the generated Moss model and desktop deployment before
applying either state. It then applies the model state, imports the model through
Moss, applies the desktop state, and runs `malm doctor`. Use `--astral` to apply
the matching Astral desktop profile.

The states intentionally remain separate. `smia-system-models` owns the
dispatcher, installer, generated model, and model helper. `default` owns the
desktop configuration and contributes the other `smia-*` plugins.

After the first apply, initialize Gnist and start the configured session:

```sh
gnist init
smia session
```

`smia session` reads the files generated by the applied profile and starts or
restarts its services.

The Hyprland profile requires Hyprland 0.55 or newer.

## Desktop profiles

In addition to the base profiles listed above, Smia includes these themed
variants:

- `mango-astral`
- `niri-astral`
- `hyprland-astral`

The installer selects one of these profiles when passed `--astral`. After
installation, switch desktop variants with the generated helper:

```sh
smia profiles switch mango-astral
```

## Everyday commands

```sh
# Check the current deployment for drift.
malm status

# Update the tracked desktop state.
malm --state default update

# Update the tracked system-model state, then review its Moss transition.
malm --state smia-system-models update
smia system-model plan

# Show the active transaction history.
malm state log

# Show resolved inputs for the active profile.
malm vars
```

Use `gnist set` to switch the active theme without applying the Malm
configuration again. For example:

```sh
gnist set aeryn
```

## Helper commands

Smia provides a unified command that discovers installed `smia-*` plugins:

| Command | What it does |
|---|---|
| `smia install MODEL` | Installs a model and its matching desktop profile. |
| `smia list` | Lists commands contributed by installed Smia plugins. |
| `smia menu` | Opens the Walker system menu. |
| `smia profiles` | Lists selectable desktop profiles. |
| `smia profiles current` | Prints the active desktop profile. |
| `smia profiles select` | Selects and applies a profile through Walker. |
| `smia profiles switch PROFILE` | Applies a profile, refreshes the session, and applies its theme. |
| `smia refresh` | Reapplies the desktop state, then refreshes the session. |
| `smia session` | Starts the graphical session target and configured services. |
| `smia system-model plan` | Previews the current generated model with Moss. |
| `smia system-model apply` | Imports the current generated model through Moss. |
| `xdg-terminal-exec [COMMAND]` | Runs a command in the terminal selected through `smia menu` defaults. |

The generated Bash configuration also provides:

| Alias | What it does |
|---|---|
| `theme` | Short alias for `gnist`. |

`smia profiles switch` skips the Malm confirmation prompt. Use `malm plan`
first when switching to an unfamiliar profile.

## Local changes

The root configuration optionally loads `~/.config/malm/local.kdl`. This keeps
machine-specific or personal changes outside the repository.

For example, a local display scale can be set with:

```kdl
variables {
    global.display-internal-scale 2.0 override=#true
}
```

Profiles can be adjusted with `extend-profile` in the same file:

```kdl
extend-profile "mango-astral" {
    use "terminals" {
        with {
            opacity 0.9
        }
    }
}
```

Run `malm plan` before applying local changes. The permission granted by
`--allow-local-includes` is recorded for later tracked updates.

System-model overrides belong in
`~/.config/malm/smia-system-models.kdl`. See
[`system-models/README.md`](system-models/README.md) for repository and package
examples.

## Working on the configuration

Clone the repository when changing modules or profiles locally:

```sh
git clone https://github.com/christian-bendiksen/smia.git
cd smia
malm check --repo . --all-profiles
malm --repo system-models --state smia-system-models check --all-profiles
malm plan --repo . --profile mango
```

Use `malm apply --repo . --profile PROFILE` only when you intentionally want to
replace the tracked remote deployment with the local checkout.

## Repository layout

```text
malm.kdl          Root Malm configuration and shared values
malm/modules/     Application and desktop modules
malm/profiles/    Base profiles and themed variants
malm/machines/    Machine-specific display configuration
system-models/    Nested bootstrap and Moss model configuration
gnist/themes/     Theme data and templates used by Gnist
```

Malm owns the deployed configuration. Most generated files come from typed
module inputs, while static files stay next to the module or profile that uses
them. Gnist updates theme files under `~/.config/gnist/themes/current` so themes
can change without rebuilding the desktop configuration.

The root `malm.kdl` currently includes `malm/machines/laptop-4k.kdl`. Change
that include when adapting the repository to another machine.

The nested system-model configuration is tracked in its own Malm state. Keeping
it in this repository lets one trusted remote provide both the system package
model and desktop configuration without combining their ownership histories.
